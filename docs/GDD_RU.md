# GDD - World of Balance / Ricochet Tanks

**Язык:** русский, основной источник правды для дизайна  
**Движок:** Unity 6  
**Активный прототип:** MainMenu -> RicochetTanks_Demo  
**Дата синхронизации:** 2026-05-01  
**Статус:** играбельный Android/PC-прототип, требуется регулярная ручная проверка после рефакторов

Этот файл — основной русский GDD. Текущая реализация и технические риски дополнительно фиксируются в `docs/TECH_STATUS.md`, план работ — в `docs/ROADMAP.md`, вопросы к геймдизайнеру — в `docs/GD_QUESTIONS.md`.

## 1. Видение

**World of Balance / Ricochet Tanks** — компактная дуэльная игра про танки с видом строго сверху.

Главная идея боя:

> Побеждает не тот, у кого больше цифры, а тот, кто лучше использует угол, позицию, рикошеты и броню.

Игрок должен читать арену как игровую доску: направление корпуса, направление башни, лазер/линию прицела, траекторию снаряда, угол встречи с броней и поверхности для рикошета.

## 2. Текущий игровой цикл

Основной flow прототипа:

```text
MainMenu
  ├── Start Game -> RicochetTanks_Demo
  ├── Statistics
  └── Exit

RicochetTanks_Demo
  ├── серия раундов до 3 побед
  ├── пауза между раундами 5 секунд
  ├── финальный результат
  ├── Restart: новая серия с 0:0
  └── Exit to MainMenu
```

Дизайн-цель текущей версии:

- быстрый вход в бой;
- короткие раунды;
- серия до 3 побед;
- понятная статистика после матчей;
- удобное mobile arcade управление;
- читаемый aim assist через лазер от башни;
- минимальный, но заметный combat feedback.

## 3. Камера и координаты

Камера для текущего прототипа — строго сверху. Не вид сбоку, не изометрия, не наклоненная кинематографическая камера.

Игровой мир построен на плоскости XZ:

- `X` — горизонтальная ось карты;
- `Z` — вертикальная ось карты;
- `Y` — только высота.

Геймплей должен оставаться на XZ:

- танки движутся по XZ;
- корпус танка вращается только вокруг Y;
- башня вращается только вокруг Y;
- снаряды летят по XZ;
- рикошет не должен вносить случайное вертикальное движение.

Рекомендуемый стиль камеры:

```text
Projection: Orthographic
Rotation:   (90, 0, 0)
```

## 4. Принципы дизайна

### 4.1. Честная физика рикошетов

Поведение снаряда должно быть читаемым и стабильным. Игрок должен понимать:

- куда летит снаряд;
- от какой поверхности он отскочит;
- сколько рикошетов уже было;
- почему снаряд пробил или не пробил;
- почему урон стал меньше после рикошета.

### 4.2. Читаемый минимализм

Прототип пока использует простую 3D-геометрию, чистые цвета, видимые снаряды, понятный HUD и минимум визуального шума. Графика служит геймплею: если эффект мешает видеть траекторию или попадание, он должен быть упрощен.

### 4.3. Угол важнее скрытых статов

Игрок должен выигрывать за счет:

- правильного позиционирования;
- наведения башни;
- использования рикошетов;
- ромбования корпуса;
- атаки в слабые зоны;
- провокации врага на плохой выстрел.

### 4.4. Малый объем, но настоящая архитектура

Даже маленький прототип должен быть сделан аккуратно:

- без God Object;
- без огромного GameManager;
- с разделением UI, геймплея и конфигов;
- с событиями;
- с отдельными модулями для снарядов, брони, здоровья, статистики, матча и ИИ;
- с возможностью расширить механику без переписывания всего проекта.

## 6. Как запустить

1. Открыть проект в Unity.
2. Для полного flow открыть `Assets/_Project/RicochetTanks/Scenes/MainMenu.unity`.
3. Нажать Play.
4. В меню нажать Start Game.
5. Проверить: управление, стрельбу, рикошеты, броню, HP, floating text, серию до 3 побед, Restart, Exit to MainMenu, Statistics.

Для быстрой проверки боя можно открыть `RicochetTanks_Demo.unity`, но основной flow для билда — через `MainMenu`.

## 7. Управление

### 7.1. ПК

```text
W / Up Arrow          ускорение вперед по корпусу
S / Down Arrow        торможение, затем задний ход
A / Left Arrow        поворот корпуса влево
D / Right Arrow       поворот корпуса вправо
Мышь                  наведение башни
ЛКМ / Space           выстрел
R                     перезапуск / restart request
```

Правило: W/S — это не движение по мировым координатам вверх/вниз. Это движение вдоль направления корпуса танка.

### 7.2. Android/mobile

Текущая mobile-схема — **arcade movement**, а не танковый W/S/A/D через стик:

```text
Левый джойстик        желаемое направление движения танка
Правый джойстик       направление башни/пушки
FIRE                  выстрел
```

Mobile-правило:

- левый стик задаёт направление в мире;
- корпус сам разворачивается к направлению стика;
- танк едет вперёд, а не включает резкий задний ход от стика назад;
- если стик тянут назад относительно танка, танк должен разворачиваться;
- правый стик управляет башней;
- отпускание стика не должно вызывать случайное вращение.

Mobile UI должен быть крупным, видимым по умолчанию и пригодным для больших пальцев.

## 8. Снаряд

У снаряда есть два разных параметра:

### 8.1. Penetration

`Penetration` отвечает на вопрос:

> Может ли снаряд пройти через эффективную броню?

### 8.2. Max Damage

`Max Damage` — это не гарантированный урон. Это верхний потолок. Реальный урон зависит от:

- эффективной брони;
- качества пробития;
- количества рикошетов до попадания;
- возможной будущей критической зоны.

## 9. Броня танка

Базовая идея:

| Зона | Смысл |
|---|---|
| Лоб | самая крепкая зона |
| Борт | пробивается при плохом угле, может держать скользящие попадания |
| Корма | слабая зона, должна наказывать за плохую позицию |

Ориентация зон:

- `+Z` локального пространства танка — Front;
- `-Z` — Rear;
- `+/-X` — Side.

## 10. Угол и effectiveArmor

Эффективная броня растет от угла попадания.

Текущая формула реализации:

```text
dot = Vector3.Dot(-incomingDirection.normalized, hitNormal.normalized)
angle = acos(clamp(dot, -1, 1))
effectiveArmor = armor / max(cos(angle), safeMinCos)
```

Дизайн-принцип:

> Чем более скользящее попадание, тем выше effectiveArmor.

## 11. Конкретные формулы боя

Текущие входные значения из конфигов:

```text
ProjectileDamage = 110
ProjectilePenetration = 45
FrontArmor = 50
SideArmor = 40
RearArmor = 10
MaxRicochets = 3
BounceSpeedMultiplier = 0.78
DamageMultiplierPerBounce = 0.75
MinProjectileSpeed = 5
```

Пробитие:

```text
if penetration < effectiveArmor:
    result = NoPenetration
    damage = 0
else:
    result = Penetrated
    damage = currentDamage
```

Auto ricochet:

```text
if hitAngle >= AutoRicochetAngle:
    result = Ricochet
    damage = 0
```

Damage после рикошета:

```text
currentDamage = currentDamage * DamageMultiplierPerBounce
```

Speed после рикошета:

```text
currentSpeed = Max(MinProjectileSpeed, currentSpeed * BounceSpeedMultiplier)
```

При текущем `BounceSpeedMultiplier = 0.78` визуальная потеря скорости может быть недостаточно заметной. Это tuning-задача, а не случайная правка.

## 12. Примеры расчета

### 12.1. Прямо в лоб

```text
FrontArmor = 50
EffectiveArmor = 50
Penetration = 45
45 < 50 -> NoPenetration / Ricochet, Damage = 0
```

### 12.2. Прямо в борт

```text
SideArmor = 40
EffectiveArmor = 40
Penetration = 45
45 >= 40 -> Penetrated
```

### 12.3. Борт под углом

```text
SideArmor = 40
EffectiveArmor > 45 из-за угла
Penetration = 45
Result = NoPenetration / Ricochet
```

### 12.4. Корма

```text
RearArmor = 10
Penetration = 45
Result = Penetrated
```

## 13. Рикошеты

Снаряд отражается от стен, препятствий и брони при непробитии/рикошете.

Отражение:

```text
Vector3.Reflect(direction, hitNormal)
```

После рикошета:

- направление остается на XZ-плоскости;
- скорость уменьшается по конфигу;
- текущий damage cap уменьшается по конфигу;
- количество оставшихся рикошетов уменьшается;
- после исчерпания лимита следующий контакт уничтожает снаряд.

## 14. Aim laser / прицеливание

Aim laser — визуальный слой помощи игроку.

Правила:

- луч начинается от muzzle/ствола;
- направление луча совпадает с направлением будущего выстрела;
- луч не должен влиять на снаряд, урон, рикошет, броню или физику;
- луч должен останавливаться на первом препятствии/танке по collision mask;
- луч не должен проходить сквозь стены, иначе он врет игроку;
- яркость/ширина должны быть конфигурируемыми и не мешать видеть поле.

Текущее состояние: луч добавлен, но требуется ручная проверка stop-on-obstacle и яркости.

## 15. Смерть танка / wreck

Желаемое поведение после смерти:

- мертвый танк остается на месте;
- управление и стрельба отключаются;
- танк становится серым/обгоревшим;
- появляется дым/искры/wreck marker;
- после паузы начинается следующий раунд или показывается финальный результат.

Текущее состояние: death/wreck VFX направление есть, но визуальный art pass и точное поведение мертвого корпуса требуют следующей итерации.

## 16. Серия раундов и карты

Текущий матч — серия до 3 побед.

Правила:

```text
RoundsToWin = 3
RoundBreakSeconds = 5
```

После раунда:

- показывается результат раунда;
- показывается счет;
- идет пауза 5 секунд;
- начинается следующий раунд.

После достижения 3 побед:

- показывается финальный результат;
- статистика сохраняется;
- доступны Restart и Exit to MainMenu.

### 16.1. Карты

Будущее направление: 3 разные карты, выбираемые случайно между раундами.

Рекомендуемая архитектура:

```text
Одна gameplay scene RicochetTanks_Demo
  └── MapRoot
        ├── Arena_01 prefab
        ├── Arena_02 prefab
        └── Arena_03 prefab
```

Лучше использовать prefab-карты внутри одной gameplay-сцены, а не отдельные сцены под каждую карту. Так проще сохранить счет серии, статистику, Android flow и будущую сетевую синхронизацию.

MVP: пока допустимо переиспользовать одну арену для всех раундов, но GDD-цель — 3 arena prefabs и random map per round.

## 17. Статистика

Статистика — важная продуктовая фича для ретеншна и тестов.

Хранение MVP:

```text
PlayerPrefs JSON
Key = RicochetTanks.PlayerStatistics.v1
```

All-time stats:

- Total matches;
- Wins;
- Losses;
- Draws;
- Win rate %;
- Shots fired;
- Hits;
- Accuracy %;
- Ricochets;
- Penetrations;
- No penetrations;
- Self hits;
- Damage dealt;
- Damage taken;
- Efficiency = DamageDealt / ShotsFired.

Recent matches:

- последние 5-10 матчей;
- результат;
- счет раундов;
- выстрелы;
- попадания;
- accuracy;
- damage dealt/taken;
- date/time, если доступно.

Правила архитектуры:

- статистика считается через `SandboxGameplayEvents`;
- статистика не живет в `HitResolver`, projectile systems, `TankHealth` или UI View;
- сохранение происходит на full session finish, не на каждый выстрел;
- UI только показывает данные из repository/service.

## 18. Enemy AI / state machine

Текущий статус: добавлен первый проход enemy AI brain/state machine, но он требует ручной проверки и следующей итерации.

Текущая желаемая роль врага:

- обнаруживать игрока;
- целиться башней в игрока;
- стрелять только при line of sight;
- не стрелять через стены;
- держать дистанцию;
- объезжать препятствия простыми raycast-решениями;
- не въезжать бесконечно в стены;
- выключаться после смерти и конца раунда.

Минимальные состояния:

```text
Idle
AimAndShoot
KeepDistance
Reposition
AvoidObstacle
Dead
```

Известный текущий issue:

```text
Враг стреляет, когда видит игрока, но может не целиться достаточно хорошо в сторону игрока.
Следующая AI-задача: проверить AimAtTarget / muzzle direction / AimAngleToShoot / line of sight и связку TurretAiming + Shooter.
```

AI не должен:

- менять формулы боя;
- создавать снаряды напрямую;
- лезть в `HitResolver`;
- использовать UI;
- ломать first-to-3 session flow и статистику.

- world-space HP bars над танками;

MainMenu должен показывать:

- Start Game;
- Statistics;

Statistics panel должен показывать all-time и recent match stats.

UI и Combat Feedback не должны:

- наносить урон;

- решать исход раунда;
- рассчитывать броню;
- напрямую зависеть от будущих PvP/network-классов.

## 21. Критическая зона / БК

Будущая идея, не текущая реализованная фича:

```text
AmmoRack / БК
```

Правила для будущего:

- если снаряд не пробил броню, крит не срабатывает;
- если снаряд пробил и попал в БК, танк уничтожается сразу;
- БК должен быть отдельной зоной/компонентом, а не случайным `if` внутри Projectile.

## 24. Network / Multiplayer

Network и multiplayer — будущая перспектива, но не ближайшая реализация.

Правило:

- не начинать network implementation до стабилизации Android-прототипа, first-to-3 flow, статистики, UI и AI MVP;
- сначала провести отдельное network architecture research;
- multiplayer prototype делать отдельной milestone-задачей.

Экспериментальный PvP/lobby UI, если присутствует в проекте, должен оставаться изолированным sandbox/mock-слоем и не подключаться к боевой сцене без отдельного решения.

- `TankHealth` — HP и смерть;
- `UI/CombatFeedback` — визуальная обратная связь;
- `Gameplay/AI` — enemy brain/state machine MVP.

## 26. Правила архитектуры

- Никаких Singleton.
- Никакого огромного `GameManager`.
- UI View только отображает данные и поднимает события.
- Presenter/Controller связывает UI с сервисами и геймплеем.
- Projectile не лезет напрямую во внутренности здоровья танка.
- Health не знает про углы и броню.
- Round/session logic отделена от здоровья и снарядов.
- Statistics logic отделена от combat и UI.
- Enemy AI управляет танком через `TankFacade`, а не через внутренности систем.
- EntryPoint/Bootstrapper только собирает зависимости.
- Приватные поля пишутся как `_camelCase`.
- Не использовать LINQ в `Update`, `FixedUpdate`, projectile systems, AI systems и горячих циклах.
- Не использовать `FindObjectOfType` / `GameObject.Find` в геймплейном цикле.
- Не класть UI-логику в gameplay systems.

## 27. Definition of Done

Фича считается готовой, если:

- она вписана в существующий модуль или создает понятный новый модуль;
- нет God Object;
- значения вынесены в конфиги;
- зависимости передаются явно;
- события подписываются и отписываются корректно;
- UI отделен от геймплея;
- горячие циклы без лишних аллокаций;
- есть достаточные debug logs;
- сцена запускается после чистого клона;
- MainMenu flow не сломан;
- Android/mobile smoke test возможен;
- README/GDD/TECH_STATUS обновлены;
- ручной чек-лист проходит или явно помечен как **Needs Manual Unity Check**.
